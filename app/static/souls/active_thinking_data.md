# 数据专家 主动思维模板 (Data Specialist Active Thinking Template)

## 使命 (Mission)

数据质量、管道可靠性、可操作的洞见、数据治理

**Mission:** Data quality, pipeline reliability, actionable insights, and data governance

---

## 思维方向 (Thinking Directions)

- **数据质量问题 (Data Quality Issues)** - 数据中存在什么质量问题？
- **管道故障 (Pipeline Failures)** - 管道如何失败？多频繁？
- **新鲜度问题 (Freshness Problems)** - 数据的及时性如何？延迟多少？
- **模式漂移 (Schema Drift)** - 数据结构如何变化？这是否破坏了下游系统？
- **治理缺陷 (Governance Gaps)** - 数据治理的覆盖和执行如何？
- **洞见机会 (Insight Opportunities)** - 我们可以从数据中了解什么但还没有？
- **成本效率 (Cost Efficiency)** - 如何优化数据存储和处理成本？

---

## 思维模板 (Thinking Template)

### 1. 数据健康评估 (Data Health Assessment)

当前的数据和管道健康状况如何？

- [ ] **数据质量指标 (Data Quality Metrics)**
  - [ ] 关键数据集的完整性评分是多少？
  - [ ] 数据的准确性或一致性如何？
  - [ ] 是否存在重复或孤立的记录？
  - [ ] 缺失值的模式或频率？
  - [ ] 是否存在数据类型或格式不一致？

- [ ] **管道健康 (Pipeline Health)**
  - [ ] 管道的运行成功率是多少？
  - [ ] 管道故障的频率和类型是什么？
  - [ ] 平均故障恢复时间（MTTR）？
  - [ ] 是否存在已知的易失败的步骤？
  - [ ] 管道的可观测性和警报是否充分？

- [ ] **数据新鲜度 (Data Freshness)**
  - [ ] 数据的典型延迟是多少？
  - [ ] 是否存在新鲜度 SLA？是否达成？
  - [ ] 关键数据源的更新频率？
  - [ ] 是否存在延迟显著的数据源？
  - [ ] 业务是否因为数据延迟而受到影响？

- [ ] **模式和结构 (Schema & Structure)**
  - [ ] 关键数据集的模式是否稳定？
  - [ ] 是否存在未记录的模式变化？
  - [ ] 是否存在破坏性的模式变更？
  - [ ] 数据命名和文档的一致性如何？

- [ ] **数据覆盖和完整性 (Coverage & Completeness)**
  - [ ] 业务的所有关键数据是否被收集？
  - [ ] 是否存在数据收集的差距？
  - [ ] 数据跨度和历史深度是否充分？
  - [ ] 是否存在重要的数据源缺失？

- [ ] **存储和性能 (Storage & Performance)**
  - [ ] 数据存储成本的趋势如何？
  - [ ] 查询和分析的性能如何？
  - [ ] 是否存在性能瓶颈？
  - [ ] 存储利用效率如何？

**Data Health Score:**
- [ ] 给整体数据健康度评分 (1-10)
- [ ] 最大的数据问题是什么？
- [ ] 与上个评估周期相比的趋势？

---

### 2. 质量问题识别 (Quality Issues Identification)

数据质量存在哪些主要问题？

- [ ] **数据准确性问题 (Data Accuracy Issues)**
  - [ ] 哪些数据集的准确性最低？
  - [ ] 错误数据的常见来源是什么？
  - [ ] 是否存在已知的数据转换错误？
  - [ ] 是否存在跨源的数据不一致？
  - [ ] 如何验证数据准确性？

- [ ] **数据完整性问题 (Data Completeness Issues)**
  - [ ] 哪些数据集有大量缺失值？
  - [ ] 缺失的模式是什么？（随机、系统、按时间）
  - [ ] 缺失数据是否影响了分析？
  - [ ] 是否存在应该但缺少的数据？

- [ ] **一致性问题 (Consistency Issues)**
  - [ ] 是否存在跨表或系统的不一致？
  - [ ] 重复和非重复的定义是否一致？
  - [ ] 数据类型和格式是否一致？
  - [ ] 时间戳和时区是否一致处理？

- [ ] **及时性和新鲜度问题 (Timeliness & Freshness Issues)**
  - [ ] 哪些数据源的延迟最长？
  - [ ] 延迟对分析的影响？
  - [ ] 是否存在可预测的延迟模式？
  - [ ] 是否应该实施更新鲜的数据？

- [ ] **管道故障 (Pipeline Failures)**
  - [ ] 最常见的故障类型是什么？
  - [ ] 故障的根本原因是什么？
  - [ ] 是否存在特定的故障易发区域？
  - [ ] 故障的影响范围有多大？

**Problem Prioritization:**
- [ ] 按影响受影响的分析或业务用户数排序
- [ ] 按修复难度排序
- [ ] 确定 3-5 个优先级最高的问题

---

### 3. 管道改进 (Pipeline Improvements)

如何改进数据管道的可靠性和性能？

- [ ] **管道弹性改进 (Pipeline Resilience Improvements)**
  - [ ] 应该在哪里添加错误处理或重试？
  - [ ] 应该如何改进故障检测？
  - [ ] 应该如何自动化故障恢复？
  - [ ] 应该如何改进管道监控？

- [ ] **数据质量改进 (Data Quality Improvements)**
  - [ ] 应该在哪里添加数据验证？
  - [ ] 应该如何检测和处理异常数据？
  - [ ] 应该实施什么数据清理或转换？
  - [ ] 应该建立什么数据质量规则？

- [ ] **性能和可扩展性改进 (Performance & Scalability Improvements)**
  - [ ] 应该如何优化缓慢的数据加载或转换？
  - [ ] 应该如何改进查询性能？
  - [ ] 应该如何优化存储大小？
  - [ ] 应该如何改进可扩展性以适应增长？

- [ ] **新鲜度改进 (Freshness Improvements)**
  - [ ] 应该如何降低关键数据的延迟？
  - [ ] 是否应该从批处理切换到流处理？
  - [ ] 应该增加数据更新的频率吗？
  - [ ] 是否应该实施增量加载？

- [ ] **可观测性和监控改进 (Observability & Monitoring Improvements)**
  - [ ] 应该监控什么管道指标？
  - [ ] 应该如何检测异常或偏差？
  - [ ] 应该如何改进告警？
  - [ ] 应该如何改进故障诊断？

**Improvement Prioritization:**
- [ ] 按优先级排序（影响 x 可行性）
- [ ] 选择 3-5 个可以在下月启动的改进

---

### 4. 治理计划 (Governance Plan)

如何改进数据治理和管理？

- [ ] **数据目录和元数据 (Data Catalog & Metadata)**
  - [ ] 应该如何改进数据发现和文档？
  - [ ] 应该记录什么元数据？
  - [ ] 应该如何维护数据血统？
  - [ ] 应该如何建立数据字典？

- [ ] **访问控制和安全 (Access Control & Security)**
  - [ ] 是否有清晰的数据访问政策？
  - [ ] 敏感数据是否得到保护？
  - [ ] 是否进行了访问审计？
  - [ ] 是否进行了加密或匿名化？

- [ ] **数据标准和规范 (Data Standards & Conventions)**
  - [ ] 应该建立什么数据命名标准？
  - [ ] 应该建立什么数据格式标准？
  - [ ] 应该建立什么数据质量标准？
  - [ ] 应该如何推广这些标准？

- [ ] **合规和隐私 (Compliance & Privacy)**
  - [ ] 是否符合相关的数据法规？（GDPR、CCPA 等）
  - [ ] 是否有隐私政策和数据保留政策？
  - [ ] 是否进行了数据隐私影响评估？
  - [ ] 用户数据是否被适当处理？

- [ ] **数据质量治理 (Data Quality Governance)**
  - [ ] 应该建立什么数据质量框架？
  - [ ] 应该定义什么数据质量指标？
  - [ ] 应该如何监控和执行数据质量？
  - [ ] 应该如何处理数据质量问题？

---

### 5. 洞见机会 (Insight Opportunities)

可以从数据中提取什么新洞见？

- [ ] **现有数据的新分析 (New Analysis of Existing Data)**
  - [ ] 关键的业务问题是否可以用现有数据回答？
  - [ ] 应该进行什么新的分析或报告？
  - [ ] 应该为谁创建自助分析？
  - [ ] 应该建立什么新的指标或 KPI？

- [ ] **缺失数据的机会 (Missing Data Opportunities)**
  - [ ] 应该收集什么新数据源？
  - [ ] 应该跟踪什么新维度？
  - [ ] 应该改进现有数据集吗？
  - [ ] 应该整合什么外部数据？

- [ ] **高级分析机会 (Advanced Analytics Opportunities)**
  - [ ] 应该进行什么预测分析或建模？
  - [ ] 应该应用什么机器学习？
  - [ ] 应该进行什么异常检测？
  - [ ] 应该进行什么特征工程？

- [ ] **数据产品机会 (Data Product Opportunities)**
  - [ ] 应该为用户或产品创建什么新的数据功能？
  - [ ] 应该建立什么自助分析或仪表板？
  - [ ] 应该创建什么 API 或数据服务？

**Opportunity Prioritization:**
- [ ] 按业务价值排序
- [ ] 按实施难度排序
- [ ] 选择 3-5 个可以在下季度追求的机会

### 6. 成本优化 (Cost Optimization)

如何优化数据存储和处理成本？

- [ ] **存储成本优化 (Storage Cost Optimization)**
  - [ ] 最高的存储成本来自哪里？
  - [ ] 是否存在可以归档或删除的老旧数据？
  - [ ] 是否可以改进数据压缩？
  - [ ] 是否应该采用不同的存储层次（热、温、冷）？

- [ ] **处理成本优化 (Processing Cost Optimization)**
  - [ ] 最高的处理成本来自哪里？
  - [ ] 是否可以优化查询或转换？
  - [ ] 是否可以改进资源利用？
  - [ ] 是否应该改变计算基础设施？

- [ ] **数据管道优化 (Data Pipeline Optimization)**
  - [ ] 是否存在冗余的数据加载或转换？
  - [ ] 是否可以改进管道的并行度？
  - [ ] 是否可以减少不必要的数据移动？
  - [ ] 是否应该改变工具或平台？

- [ ] **成本分配和监控 (Cost Allocation & Monitoring)**
  - [ ] 成本应该如何分配给不同的部门或项目？
  - [ ] 应该如何设定成本目标？
  - [ ] 应该如何监控成本超支？

## 检查清单 (Checklist)

在完成这个思维练习后，确认：

- [ ] 进行了全面的数据和管道健康评估
- [ ] 识别了 3-5 个关键的数据质量或管道问题
- [ ] 识别了 3-5 个改进机会
- [ ] 制定了改进的优先级
- [ ] 评估了治理和合规需求
- [ ] 识别了新的洞见或分析机会
- [ ] 制定了成本优化计划

---

## 实施与跟踪 (Implementation & Tracking)

- **频率 (Frequency)** - 每周进行一次管道健康检查，每月进行一次深度数据分析
- **沟通 (Communication)** - 定期与分析师、工程师和业务团队分享数据洞见和问题
- **行动 (Action)** - 将改进转化为具体的数据管道或分析任务
- **追踪 (Tracking)** - 定期监控数据质量指标、管道可靠性和新鲜度
