# 编码员 主动思维模板 (Coder Active Thinking Template)

## 使命 (Mission)

代码质量、更少 bug、系统稳定性、开发效率

**Mission:** Code quality, fewer bugs, system stability, and development efficiency

---

## 思维方向 (Thinking Directions)

- **容易出 bug 的区域 (Bug-Prone Areas)** - 哪些代码容易产生 bug？模式是什么？
- **架构耦合 (Architecture Coupling)** - 哪些组件耦合过度？这是否影响了开发速度？
- **代码可读性 (Code Readability)** - 代码是否容易理解？是否有改进空间？
- **测试覆盖缺陷 (Testing Gaps)** - 测试覆盖率最低的地方在哪？为什么？
- **自动化机会 (Automation Opportunities)** - 哪些重复的工作可以自动化？
- **事故防止 (Incident Prevention)** - 最近的事故能否避免？如何防止类似事故？
- **技能缺口 (Skill Gaps)** - 我在哪些技能上的进步最慢？如何改进？

---

## 思维模板 (Thinking Template)

### 1. 代码与系统问题 (Code & System Issues)

当前在代码质量和系统稳定性方面存在什么问题？

- [ ] **性能问题 (Performance Issues)**
  - [ ] 最近是否发现了性能回归？
  - [ ] 是否存在已知的慢查询或低效操作？
  - [ ] 是否有容易改进的优化机会？
  - [ ] 是否对代码进行了基准测试？

- [ ] **错误和 Bug (Errors & Bugs)**
  - [ ] 最近修复了哪些 bug？模式是什么？
  - [ ] 是否存在重复出现的 bug 类别？
  - [ ] 哪些功能的故障率最高？
  - [ ] 用户报告了哪些问题？

- [ ] **代码质量指标 (Code Quality Metrics)**
  - [ ] 代码覆盖率是多少？（总体和关键路径）
  - [ ] 是否有明显的代码重复？
  - [ ] 循环复杂度是否过高？
  - [ ] 是否有长函数或类？

- [ ] **依赖和库 (Dependencies & Libraries)**
  - [ ] 是否有过期的依赖项？
  - [ ] 是否有安全漏洞尚未修补？
  - [ ] 是否有冗余或不必要的依赖项？
  - [ ] 依赖项是否有破坏性更新？

- [ ] **最近的工作问题 (Recent Work Issues)**
  - [ ] 上周我遇到的最大障碍是什么？
  - [ ] 是否有代码审查反馈要求改进？
  - [ ] 是否有技术上困难的任务？
  - [ ] 是否花费了太多时间调试问题？

**Problem Severity Assessment:**
- [ ] 哪个问题对用户影响最大？
- [ ] 哪个问题影响了团队的生产力？
- [ ] 哪个问题最容易修复？

---

### 2. 技术风险 (Technical Risks)

在代码和架构中存在什么技术风险？

- [ ] **架构和设计风险 (Architecture & Design Risks)**
  - [ ] 是否存在过度耦合的模块？
  - [ ] 是否有明确的关注点分离？
  - [ ] 是否存在循环依赖？
  - [ ] 是否有临时性解决方案（hack）？

- [ ] **测试覆盖风险 (Test Coverage Risks)**
  - [ ] 关键功能的测试覆盖率是多少？
  - [ ] 是否有测试覆盖的盲点？
  - [ ] 是否有无法轻易测试的代码？
  - [ ] 边界情况是否被充分测试？

- [ ] **维护性风险 (Maintainability Risks)**
  - [ ] 是否有大型代码块难以理解？
  - [ ] 是否有不一致的命名或编码风格？
  - [ ] 文档是否与代码同步？
  - [ ] 是否有大量的"魔法数字"或常数硬编码？

- [ ] **扩展性和适应性风险 (Scalability & Adaptability Risks)**
  - [ ] 代码是否容易扩展新功能？
  - [ ] 是否有难以修改的假设硬编码？
  - [ ] 是否存在可能阻止未来重构的决策？
  - [ ] 是否有性能优化会降低可维护性？

- [ ] **事故和恢复风险 (Incident & Recovery Risks)**
  - [ ] 最近的事故从何而来？
  - [ ] 是否有"已知问题"（Known Issues）？
  - [ ] 是否有可能导致数据丢失的代码？
  - [ ] 恢复时间是否因为诊断困难而增加？

**Risk Prioritization:**
- [ ] 按优先级排序最严重的 3-5 个风险
- [ ] 对每个风险的缓解策略是什么？

---

### 3. 优化计划 (Optimization Plan)

代码和系统有哪些改进机会？

- [ ] **性能优化 (Performance Optimization)**
  - [ ] 哪个函数或操作最慢？
  - [ ] 可以采用什么具体的优化策略？（缓存、索引、并行化等）
  - [ ] 优化的预期收益是什么？
  - [ ] 实施成本和复杂性如何？

- [ ] **代码简化 (Code Simplification)**
  - [ ] 是否有可以删除的冗余代码？
  - [ ] 是否有可以提取为单独函数的常见模式？
  - [ ] 是否可以使用更清晰的算法或数据结构？
  - [ ] 是否有可以替换为库的实现？

- [ ] **测试覆盖改进 (Test Coverage Improvement)**
  - [ ] 优先为哪些区域添加测试？
  - [ ] 是否可以改进现有测试的质量？
  - [ ] 是否应该添加集成或端到端测试？
  - [ ] 是否应该引入性能或压力测试？

- [ ] **重构机会 (Refactoring Opportunities)**
  - [ ] 哪些类或模块应该重构？
  - [ ] 应该提取哪些公共模式？
  - [ ] 是否应该采用不同的设计模式？
  - [ ] 是否应该重新组织代码库结构？

- [ ] **自动化机会 (Automation Opportunities)**
  - [ ] 哪些手工流程可以自动化？
  - [ ] 是否可以改进 CI/CD 流程？
  - [ ] 是否可以自动化代码审查检查？
  - [ ] 是否可以自动化部署前检查？

- [ ] **技术债清理 (Tech Debt Cleanup)**
  - [ ] 最紧迫的技术债是什么？
  - [ ] 清理这些债务的计划是什么？
  - [ ] 优先级和时间表如何？
  - [ ] 对代码库和开发速度的预期影响？

**Opportunity Prioritization:**
- [ ] 按收益/成本比排序
- [ ] 选择 3-5 个可以在下周/下月完成的项目

---

### 4. 实施步骤 (Implementation Steps)

具体的实施计划是什么？

- [ ] **具体任务 (Specific Tasks)**
  - [ ] 优先级任务列表（按顺序）
  - [ ] 每个任务的估计工作量
  - [ ] 依赖性和阻塞点

- [ ] **测试策略 (Testing Strategy)**
  - [ ] 如何验证改进的正确性？
  - [ ] 需要进行哪些测试？
  - [ ] 是否需要 A/B 测试或渐进式部署？

- [ ] **验收标准 (Acceptance Criteria)**
  - [ ] 成功的定义是什么？
  - [ ] 需要达到哪些指标？
  - [ ] 如何衡量改进？

- [ ] **审查和反馈 (Review & Feedback)**
  - [ ] 代码审查计划是什么？
  - [ ] 应该寻求谁的审查？
  - [ ] 是否需要 TDD（测试驱动开发）？

---

### 5. 自测与验证 (Self-Testing & Verification)

如何确保改进的质量？

- [ ] **单元测试 (Unit Tests)**
  - [ ] 新代码是否有适当的单元测试？
  - [ ] 测试覆盖率是否达到目标（如80%+）？
  - [ ] 是否有边界情况测试？

- [ ] **集成测试 (Integration Tests)**
  - [ ] 是否测试了与其他模块的交互？
  - [ ] 是否测试了数据库集成？
  - [ ] 是否测试了 API 调用？

- [ ] **手动测试 (Manual Testing)**
  - [ ] 应该手动测试哪些场景？
  - [ ] 是否需要测试各种浏览器/设备？
  - [ ] 是否需要性能或压力测试？

- [ ] **代码审查检查清单 (Code Review Checklist)**
  - [ ] 是否遵循项目的编码标准？
  - [ ] 是否有明确的变量和函数名？
  - [ ] 是否添加了必要的注释或文档？
  - [ ] 是否存在潜在的 bug 或边界情况？

---

### 6. 能力提升 (Capability Improvement)

作为编码者，我如何改进自己？

- [ ] **技术技能缺口 (Technical Skill Gaps)**
  - [ ] 我在哪些编程语言或框架上最弱？
  - [ ] 有什么设计模式我还没掌握？
  - [ ] 我的系统设计知识如何？
  - [ ] 我是否需要改进数据库或算法知识？

- [ ] **学习计划 (Learning Plan)**
  - [ ] 下个月我应该学习什么？
  - [ ] 是否有有用的文章、书籍或课程？
  - [ ] 是否可以通过代码审查来学习？
  - [ ] 是否应该寻求导师或配对编程？

- [ ] **工作方式改进 (Work Style Improvement)**
  - [ ] 我的开发流程是否高效？
  - [ ] 是否有工具可以加快我的工作？
  - [ ] 是否花费了太多时间调试而不是设计？
  - [ ] 是否应该采用不同的工作方法？

- [ ] **协作和沟通 (Collaboration & Communication)**
  - [ ] 我是否清楚地表达了我的代码意图？
  - [ ] 我对代码审查反馈的反应如何？
  - [ ] 我是否帮助他人改进代码？
  - [ ] 我是否记录了我的设计决策？

---

### 7. 自我反思 (Self-Reflection)

- [ ] **分析质量 (Quality of Analysis)**
  - [ ] 我是否识别了所有主要的代码问题？
  - [ ] 我的优先级是否合理？
  - [ ] 我是否高估或低估了改进的难度？

- [ ] **执行力 (Execution Reality)**
  - [ ] 我是否能在本周完成计划中的项目？
  - [ ] 是否有外部依赖或阻塞？
  - [ ] 是否有我低估的复杂性？

- [ ] **学习 (Learning)**
  - [ ] 上周我学到了什么？
  - [ ] 这次思维练习是否改变了我的方法？
  - [ ] 我在编码实践上是否看到了进展？

- [ ] **下一步 (Next Steps)**
  - [ ] 明天我应该首先处理什么？
  - [ ] 我需要与谁讨论我的计划？
  - [ ] 是否有我需要先研究的东西？

---

## 检查清单 (Checklist)

在完成这个思维练习后，确认：

- [ ] 识别了 3-5 个最关键的代码或系统问题
- [ ] 识别了 3-5 个优先级改进机会
- [ ] 为改进制定了具体的实施步骤
- [ ] 定义了成功的衡量标准
- [ ] 考虑了测试策略
- [ ] 识别了能力提升领域
- [ ] 规划了下周的工作重点

---

## 实施与跟踪 (Implementation & Tracking)

- **频率 (Frequency)** - 每周一次（通常在周末或周一），特别是在完成大型任务或遇到问题后进行加强
- **沟通 (Communication)** - 在代码审查或团队讨论中分享关键发现
- **行动 (Action)** - 将改进转化为具体的任务或 story
- **追踪 (Tracking)** - 每日检查任务进度，每周回顾计划的执行情况
