# 架构师 主动思维模板 (Architect Active Thinking Template)

## 使命 (Mission)

系统健康、可扩展性、可维护性、面向未来

**Mission:** System health, scalability, maintainability, and future-proofing

---

## 思维方向 (Thinking Directions)

- **架构债 (Architecture Debt)** - 当前架构有哪些主要限制或不足？
- **扩展瓶颈 (Scaling Bottlenecks)** - 什么会阻止我们扩展？
- **服务边界 (Service Boundaries)** - 当前的服务和模块是否有适当的边界？
- **数据流效率 (Data Flow Efficiency)** - 数据如何流动？是否有改进空间？
- **灾难恢复 (Disaster Recovery)** - 我们对故障的准备有多充分？
- **技术演进 (Technology Evolution)** - 架构如何适应新技术和需求？

---

## 思维模板 (Thinking Template)

### 1. 系统健康评估 (System Health Assessment)

当前架构的整体健康状况如何？

- [ ] **架构概览 (Architecture Overview)**
  - [ ] 当前的主要组件和服务有哪些？
  - [ ] 它们之间的关系和依赖性是什么？
  - [ ] 是单体架构、微服务还是混合架构？
  - [ ] 关键的设计模式和原则是什么？
  - [ ] 是否遵循一致的架构模式？

- [ ] **可扩展性指标 (Scalability Metrics)**
  - [ ] 系统目前支持多少用户/请求？
  - [ ] 可以扩展到什么程度？
  - [ ] 扩展过程中的主要障碍是什么？
  - [ ] 垂直扩展 vs 水平扩展的权衡是什么？
  - [ ] 数据库、缓存、消息队列的扩展能力如何？

- [ ] **可维护性评估 (Maintainability Assessment)**
  - [ ] 架构是否容易理解？
  - [ ] 新开发者入职需要多长时间？
  - [ ] 模块之间的耦合程度如何？
  - [ ] 是否存在单点故障或关键依赖？
  - [ ] 代码库的组织是否合理？

- [ ] **性能特征 (Performance Characteristics)**
  - [ ] 不同服务的延迟特征如何？
  - [ ] 是否存在已知的性能瓶颈？
  - [ ] 吞吐量如何分布在不同的组件中？
  - [ ] 资源利用效率如何？
  - [ ] 是否存在冗余或浪费？

- [ ] **可靠性和容错性 (Reliability & Fault Tolerance)**
  - [ ] 系统的冗余程度如何？
  - [ ] 故障点（SPOF）在哪里？
  - [ ] 故障恢复策略是否充分？
  - [ ] 是否实施了断路器、重试等模式？
  - [ ] 灾难恢复计划是什么？

**Health Scoring:**
- [ ] 给整体架构健康度评分 (1-10)
- [ ] 最大的架构拖累因素是什么？
- [ ] 与理想架构相比的差距有多大？

### 2. 架构风险 (Architecture Risks)

架构和系统设计中存在哪些主要风险？

- [ ] **扩展性风险 (Scalability Risks)**
  - [ ] 当负载增加 10 倍时会发生什么？
  - [ ] 是否存在不能线性扩展的组件？
  - [ ] 数据库是否会成为瓶颈？
  - [ ] 是否需要分片或分区策略？
  - [ ] 缓存策略是否充分？

- [ ] **可维护性风险 (Maintainability Risks)**
  - [ ] 架构是否太复杂？
  - [ ] 开发者是否难以理解或修改系统？
  - [ ] 是否存在架构规范漂移？
  - [ ] 是否积累了太多的设计债务？
  - [ ] 是否存在隐藏的依赖或耦合？

- [ ] **可靠性风险 (Reliability Risks)**
  - [ ] 是否存在级联故障的风险？
  - [ ] 主要依赖项的可靠性如何？
  - [ ] 故障检测和恢复是否充分？
  - [ ] 是否有数据一致性风险？
  - [ ] 是否有备份和恢复的缺陷？

- [ ] **安全性风险 (Security Risks)**
  - [ ] 服务之间的通信是否安全？
  - [ ] 是否存在数据隔离问题？
  - [ ] 访问控制是否充分？
  - [ ] 是否有供应链安全风险？
  - [ ] 是否有已知的架构级别的安全漏洞？

- [ ] **技术风险 (Technology Risks)**
  - [ ] 使用的技术是否过时或即将被淘汰？
  - [ ] 是否存在技术碎片化？
  - [ ] 是否太依赖特定的供应商或技术？
  - [ ] 迁移到新技术的成本有多高？
  - [ ] 是否存在技术堆栈不匹配？

- [ ] **成本风险 (Cost Risks)**
  - [ ] 基础设施成本是否过高？
  - [ ] 是否存在资源浪费？
  - [ ] 是否可以优化成本而不影响性能？
  - [ ] 是否有成本预测和控制？

**Risk Prioritization:**
- [ ] 按影响 x 可能性排序风险
- [ ] 确定 3-5 个优先级最高的风险
- [ ] 对每个风险的缓解策略是什么？

### 3. 设计改进 (Design Improvements)

有哪些主要的架构改进机会？

- [ ] **服务和模块设计 (Service & Module Design)**
  - [ ] 是否应该分解某些大型模块？
  - [ ] 是否应该整合某些小型服务？
  - [ ] 是否应该改变服务边界？
  - [ ] 是否应该采用新的通信模式（同步、异步）？
  - [ ] 是否应该采用特定的设计模式（saga、event sourcing 等）？

- [ ] **数据架构改进 (Data Architecture Improvements)**
  - [ ] 是否应该改变数据库选择或架构？
  - [ ] 是否应该采用读写分离（CQRS）？
  - [ ] 是否应该改进缓存策略？
  - [ ] 是否应该采用数据湖或数据仓库？
  - [ ] 是否应该改进数据一致性模型？

- [ ] **API 和集成改进 (API & Integration Improvements)**
  - [ ] API 设计是否合理？
  - [ ] 是否应该采用 API 网关？
  - [ ] 是否应该改进版本管理策略？
  - [ ] 是否应该标准化集成模式？
  - [ ] 是否应该改进错误处理和重试？

- [ ] **可观测性改进 (Observability Improvements)**
  - [ ] 监控覆盖是否充分？
  - [ ] 日志策略是否足够？
  - [ ] 是否有分布式追踪？
  - [ ] 告警是否及时有效？
  - [ ] 是否可以改进故障诊断能力？

- [ ] **部署和运维改进 (Deployment & Operations Improvements)**
  - [ ] 部署过程是否可以自动化？
  - [ ] 是否应该采用容器化或 Kubernetes？
  - [ ] 是否应该改进 CI/CD 流程？
  - [ ] 是否应该采用基础设施即代码？
  - [ ] 是否可以改进配置管理？

- [ ] **安全架构改进 (Security Architecture Improvements)**
  - [ ] 是否应该采用零信任架构？
  - [ ] 是否应该改进身份验证和授权？
  - [ ] 是否应该加强网络隔离？
  - [ ] 是否应该改进密钥管理？
  - [ ] 是否应该采用安全扫描和测试？

**Improvement Prioritization:**
- [ ] 按优先级排序（影响 x 可行性）
- [ ] 选择 3-5 个可以在下季度启动的改进

### 4. 迁移计划 (Migration Plan)

如何从当前架构演变到目标架构？

- [ ] **目标架构定义 (Target Architecture Definition)**
  - [ ] 目标架构应该是什么样的？
  - [ ] 关键的架构决策是什么？
  - [ ] 与当前架构的主要区别是什么？
  - [ ] 目标架构何时应该实现？

- [ ] **迁移策略 (Migration Strategy)**
  - [ ] 应该采用大爆炸迁移还是渐进式迁移？
  - [ ] 应该如何分阶段？
  - [ ] 不同阶段的优先级是什么？
  - [ ] 如何处理向后兼容性？
  - [ ] 如何处理数据迁移？

- [ ] **风险和缓解 (Risks & Mitigation)**
  - [ ] 迁移过程中的关键风险是什么？
  - [ ] 如何测试迁移？
  - [ ] 如何进行回滚？
  - [ ] 如何最小化对用户的影响？
  - [ ] 如何监控迁移进度？

- [ ] **资源和时间表 (Resources & Timeline)**
  - [ ] 迁移需要什么资源？
  - [ ] 时间表应该是什么？
  - [ ] 里程碑和检查点是什么？
  - [ ] 可能的阻塞点或依赖是什么？

- [ ] **团队和沟通 (Team & Communication)**
  - [ ] 谁拥有迁移？
  - [ ] 如何与各个团队协调？
  - [ ] 如何保持利益相关者的了解？
  - [ ] 如何处理外部依赖或合作伙伴？

### 5. 文档 (Documentation)

如何记录架构？

- [ ] **架构决策记录 (Architecture Decision Records - ADR)**
  - [ ] 是否应该创建或更新 ADR？
  - [ ] 哪些关键决策应该被记录？
  - [ ] 如何组织和维护 ADR？

- [ ] **架构视图和图表 (Architecture Views & Diagrams)**
  - [ ] 是否应该更新高级架构图？
  - [ ] 是否应该创建详细的组件图？
  - [ ] 是否应该创建数据流图？
  - [ ] 是否应该创建部署图？

- [ ] **设计指南和最佳实践 (Design Guidelines & Best Practices)**
  - [ ] 是否应该创建或更新设计指南？
  - [ ] 应该涵盖哪些领域？（服务设计、数据设计、API 设计等）
  - [ ] 如何推广这些指南？
  - [ ] 如何保持指南的最新？

- [ ] **可运维性文档 (Operability Documentation)**
  - [ ] 是否应该创建运维指南？
  - [ ] 应该记录常见问题和解决方案吗？
  - [ ] 应该记录部署流程吗？
  - [ ] 应该记录故障诊断过程吗？

---

### 6. 团队对齐 (Team Alignment)

如何确保团队理解和遵循架构？

- [ ] **架构沟通 (Architecture Communication)**
  - [ ] 应该进行什么级别的架构培训？
  - [ ] 应该如何定期审查架构？
  - [ ] 应该如何收集和处理架构反馈？
  - [ ] 应该建立什么架构治理？

- [ ] **架构评审 (Architecture Reviews)**
  - [ ] 应该如何进行架构评审？
  - [ ] 应该检查什么？（遵从性、性能影响等）
  - [ ] 谁应该参与？
  - [ ] 审查流程应该是什么？

- [ ] **知识分享 (Knowledge Sharing)**
  - [ ] 应该进行什么架构讲座或工作坊？
  - [ ] 应该创建什么内部资源或维基？
  - [ ] 应该如何引导新的架构师或高级工程师？
  - [ ] 应该进行什么同行审查或配对编程？

- [ ] **团队能力 (Team Capability)**
  - [ ] 团队是否有架构设计所需的技能？
  - [ ] 是否需要培训或招聘？
  - [ ] 是否应该引入外部顾问？
  - [ ] 应该如何建立架构领导力？

### 7. 自我反思 (Self-Reflection)

- [ ] **架构分析质量 (Quality of Architecture Analysis)**
  - [ ] 我是否充分考虑了所有关键的架构方面？
  - [ ] 我的假设是否有依据？
  - [ ] 我是否过度或不足地强调了某些因素？
  - [ ] 我的建议是否现实可行？

- [ ] **未来视角 (Future Perspective)**
  - [ ] 我是否考虑了 3-5 年的增长和变化？
  - [ ] 我是否考虑了新兴技术的潜在影响？
  - [ ] 我是否考虑了市场和监管变化？
  - [ ] 我的架构是否足够灵活以适应未来变化？

- [ ] **权衡和平衡 (Trade-offs & Balance)**
  - [ ] 我是否平衡了性能、成本和复杂性？
  - [ ] 我是否平衡了创新和稳定性？
  - [ ] 我是否平衡了短期需求和长期愿景？
  - [ ] 我是否充分考虑了团队能力和组织约束？

- [ ] **盲点和学习 (Blind Spots & Learning)**
  - [ ] 我可能忽视了什么重要的架构方面？
  - [ ] 是否有我应该寻求专家意见的领域？
  - [ ] 我从其他架构或竞争对手那里学到了什么？
  - [ ] 有什么架构知识我应该更新或深化？

- [ ] **下一步 (Next Steps)**
  - [ ] 最关键的架构行动项是什么？
  - [ ] 应该与谁讨论这些改进？
  - [ ] 应该如何启动迁移计划？
  - [ ] 应该如何衡量改进的成功？

## 检查清单 (Checklist)

在完成这个思维练习后，确认：

- [ ] 进行了全面的架构评估
- [ ] 识别了 3-5 个关键的架构风险
- [ ] 识别了 3-5 个主要的改进机会
- [ ] 制定了清晰的目标架构愿景
- [ ] 制定了可行的迁移计划
- [ ] 确定了文档和沟通需求
- [ ] 规划了团队能力建设

---

## 实施与跟踪 (Implementation & Tracking)

- **频率 (Frequency)** - 每季度进行一次深度架构审查，每月进行一次架构设计讨论
- **沟通 (Communication)** - 定期向技术领导和团队分享架构洞见
- **行动 (Action)** - 将架构决策转化为具体的工程任务和 epic
- **追踪 (Tracking)** - 监控架构改进的进度，定期审查关键指标
