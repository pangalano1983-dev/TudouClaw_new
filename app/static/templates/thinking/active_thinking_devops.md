# DevOps 主动思维模板 (DevOps Active Thinking Template)

## 使命 (Mission)

可靠性、正常运行时间、部署速度、安全性、成本优化

**Mission:** Reliability, uptime, deployment velocity, security, and cost optimization

---

## 思维方向 (Thinking Directions)

- **事故模式 (Incident Patterns)** - 最近发生了什么故障？存在什么模式？
- **部署风险 (Deployment Risks)** - 部署过程中的主要风险是什么？
- **监控缺陷 (Monitoring Gaps)** - 监控覆盖中存在什么空白？
- **安全漏洞 (Security Vulnerabilities)** - 基础设施中存在什么安全问题？
- **成本浪费 (Cost Waste)** - 如何优化成本？
- **自动化机会 (Automation Opportunities)** - 什么可以自动化？
- **容量规划 (Capacity Planning)** - 基础设施是否能支持增长？

---

## 思维模板 (Thinking Template)

### 1. 基础设施状态 (Infrastructure Status)

当前的基础设施和运维状况如何？

- [ ] **正常运行时间和可靠性 (Uptime & Reliability)**
  - [ ] 最近的正常运行时间百分比是多少？
  - [ ] 是否达到了 SLA 目标？
  - [ ] 有哪些计划外的停机？
  - [ ] 最长的故障是什么时候发生的？
  - [ ] 故障恢复时间（MTTR）的趋势如何？

- [ ] **部署频率和速度 (Deployment Frequency & Speed)**
  - [ ] 每天/周/月进行多少次部署？
  - [ ] 从提交到生产的平均时间？
  - [ ] 部署失败率是多少？
  - [ ] 部署回滚的频率？
  - [ ] 部署过程中的主要瓶颈在哪里？

- [ ] **监控和可观测性 (Monitoring & Observability)**
  - [ ] 监控覆盖程度如何（百分比）？
  - [ ] 是否能检测到所有关键故障？
  - [ ] 告警是否及时且有用？
  - [ ] 日志和指标的完整性如何？
  - [ ] 故障诊断的难度如何？

- [ ] **安全态势 (Security Posture)**
  - [ ] 安全扫描和审计的结果如何？
  - [ ] 是否存在已知的安全漏洞？
  - [ ] 是否有未打补丁的系统？
  - [ ] 是否存在访问控制问题？
  - [ ] 是否有数据保护问题？

- [ ] **成本和资源利用 (Cost & Resource Utilization)**
  - [ ] 当前的月度 / 年度成本？
  - [ ] 成本的趋势如何（上升、稳定、下降）？
  - [ ] 哪些区域成本最高？
  - [ ] 资源利用效率如何？
  - [ ] 是否存在明显的浪费？

- [ ] **容量和扩展 (Capacity & Scaling)**
  - [ ] 当前的容量用量百分比？
  - [ ] 增长率如何？
  - [ ] 什么时候会达到容量限制？
  - [ ] 扩展过程有多快？
  - [ ] 是否有自动扩展？

**Infrastructure Health Score:**
- [ ] 给整体基础设施健康度评分 (1-10)
- [ ] 最大的运维问题是什么？
- [ ] 与上个评估周期相比的变化趋势？

---

### 2. 可靠性风险 (Reliability Risks)

基础设施和部署中存在哪些风险？

- [ ] **故障模式风险 (Failure Pattern Risks)**
  - [ ] 最常见的故障类型是什么？
  - [ ] 是否存在已知的级联故障风险？
  - [ ] 主要的单点故障在哪里？
  - [ ] 是否有部分故障检测？
  - [ ] 故障恢复自动化程度如何？

- [ ] **部署风险 (Deployment Risks)**
  - [ ] 部署过程中的最大风险是什么？
  - [ ] 是否存在部署可能导致数据丢失的情况？
  - [ ] 部署是否可以回滚？
  - [ ] 是否有零停机部署？
  - [ ] 是否充分测试了部署流程？

- [ ] **依赖性风险 (Dependency Risks)**
  - [ ] 外部依赖的可靠性如何？
  - [ ] 是否有降级或故障转移策略？
  - [ ] 依赖项是否经常改变或不稳定？
  - [ ] 是否能检测到依赖项故障？

- [ ] **数据和备份风险 (Data & Backup Risks)**
  - [ ] 备份策略是什么？
  - [ ] 备份多久测试一次？
  - [ ] 恢复过程有多复杂？
  - [ ] 恢复时间目标（RTO）和恢复点目标（RPO）是什么？
  - [ ] 是否有数据丢失的风险？

- [ ] **安全风险 (Security Risks)**
  - [ ] 是否有已知的安全漏洞？
  - [ ] 补丁应用流程如何？
  - [ ] 是否存在访问控制问题？
  - [ ] 是否有合规问题？
  - [ ] 是否存在供应链安全风险？

- [ ] **配置和漂移风险 (Configuration & Drift Risks)**
  - [ ] 基础设施是否处于已知的期望状态？
  - [ ] 是否检测到配置漂移？
  - [ ] 是否存在手动配置变化？
  - [ ] 是否有版本控制的所有配置？

**Risk Prioritization:**
- [ ] 按影响 x 可能性排序风险
- [ ] 确定 3-5 个优先级最高的风险
- [ ] 对每个风险的缓解策略是什么？

---

### 3. 改进计划 (Improvement Plan)

如何改进可靠性、速度、安全性和成本？

- [ ] **可靠性改进 (Reliability Improvements)**
  - [ ] 如何消除单点故障？
  - [ ] 是否应该添加冗余？
  - [ ] 是否应该改进故障检测？
  - [ ] 是否应该自动化故障恢复？
  - [ ] 是否应该改进灾难恢复？

- [ ] **部署速度改进 (Deployment Speed Improvements)**
  - [ ] 部署流程的瓶颈在哪里？
  - [ ] 什么可以并行化？
  - [ ] 什么可以自动化？
  - [ ] 测试过程可以加快吗？
  - [ ] 是否应该采用蓝绿部署或金丝雀部署？

- [ ] **监控和可观测性改进 (Monitoring & Observability Improvements)**
  - [ ] 监控覆盖中的最大缺陷？
  - [ ] 应该添加什么指标或日志？
  - [ ] 告警策略是否应该改进？
  - [ ] 是否需要分布式追踪？
  - [ ] 是否应该改进仪表板？

- [ ] **安全改进 (Security Improvements)**
  - [ ] 应该如何加固基础设施？
  - [ ] 是否应该采用零信任架构？
  - [ ] 应该改进补丁管理吗？
  - [ ] 是否应该进行安全审计或渗透测试？
  - [ ] 应该改进密钥管理吗？

- [ ] **成本优化 (Cost Optimization)**
  - [ ] 最大的浪费区域在哪里？
  - [ ] 是否可以优化实例大小？
  - [ ] 是否可以使用预留实例或现货实例？
  - [ ] 是否存在无用的资源可以删除？
  - [ ] 是否可以改进数据存储成本？

- [ ] **自动化机会 (Automation Opportunities)**
  - [ ] 哪些手工流程可以自动化？
  - [ ] 是否应该采用基础设施即代码？
  - [ ] 是否应该改进配置管理？
  - [ ] 是否应该自动化安全扫描？
  - [ ] 是否应该自动化容量规划？

**Improvement Prioritization:**
- [ ] 按优先级排序（影响 x 可行性）
- [ ] 选择 3-5 个可以在下月启动的改进

---

### 4. 自动化优先级 (Automation Priorities)

什么应该首先自动化？

- [ ] **高优先级自动化 (High-Priority Automation)**
  - [ ] 频繁的、容易出错的手工流程
  - [ ] 关键的、时间敏感的流程
  - [ ] 重复的、没有增加价值的流程
  - [ ] 具有明显 ROI 的自动化

- [ ] **自动化实施计划 (Automation Implementation Plan)**
  - [ ] 优先级顺序和时间表
  - [ ] 每个项目的投入和预期节省
  - [ ] 风险和缓解策略
  - [ ] 如何验证自动化的正确性

- [ ] **工具和平台 (Tools & Platforms)**
  - [ ] 应该使用什么工具或平台？
  - [ ] 现有工具是否可以改进或扩展？
  - [ ] 是否需要采用新的工具？
  - [ ] 成本和学习曲线如何？

---

### 5. 成本优化 (Cost Optimization)

如何降低基础设施成本？

- [ ] **成本分析 (Cost Analysis)**
  - [ ] 按服务或部门分解成本
  - [ ] 识别成本异常或突增
  - [ ] 将成本与使用或收入相关联
  - [ ] 与基准进行比较

- [ ] **优化策略 (Optimization Strategies)**
  - [ ] 实例大小和类型优化
  - [ ] 预留容量采购
  - [ ] 现货实例使用
  - [ ] 数据存储优化
  - [ ] 网络成本优化

- [ ] **预算和监控 (Budgeting & Monitoring)**
  - [ ] 应该设定什么成本目标？
  - [ ] 如何跟踪成本与预算？
  - [ ] 应该如何分配成本？
  - [ ] 应该如何处理成本超支？

---

### 6. 安全态势 (Security Posture)

基础设施的安全性如何？

- [ ] **安全扫描和测试 (Security Scanning & Testing)**
  - [ ] 应该进行什么定期的安全扫描？
  - [ ] 应该进行什么渗透测试？
  - [ ] 应该如何处理发现的问题？
  - [ ] 是否有漏洞修复的 SLA？

- [ ] **访问控制和身份 (Access Control & Identity)**
  - [ ] 是否正确实施了 RBAC？
  - [ ] 是否使用了多因素认证？
  - [ ] 是否审计了访问日志？
  - [ ] 离职员工的访问权限是否及时移除？

- [ ] **加密和密钥管理 (Encryption & Key Management)**
  - [ ] 传输中的数据是否加密？
  - [ ] 静止的数据是否加密？
  - [ ] 密钥管理是否安全？
  - [ ] 是否进行了密钥轮换？

- [ ] **合规和审计 (Compliance & Audit)**
  - [ ] 是否符合必要的合规标准？（HIPAA、GDPR 等）
  - [ ] 是否进行了审计？
  - [ ] 是否记录了所有关键操作？
  - [ ] 是否有审计追踪？

### 7. 自我反思 (Self-Reflection)

- [ ] **运维策略质量 (Quality of Operations Strategy)**
  - [ ] 我是否充分考虑了所有关键的运维方面？
  - [ ] 我的优先级是否合理？
  - [ ] 我是否过度或不足地强调了某些因素？

- [ ] **风险评估准确性 (Risk Assessment Accuracy)**
  - [ ] 我是否正确识别了最关键的风险？
  - [ ] 我是否遗漏了任何重要的风险？
  - [ ] 我的缓解策略是否充分？

- [ ] **执行准备 (Execution Readiness)**
  - [ ] 我的改进计划是否现实可行？
  - [ ] 是否有足够的资源来执行？
  - [ ] 是否有外部依赖或阻塞？

- [ ] **学习和改进 (Learning & Improvement)**
  - [ ] 最近的故障中学到了什么？
  - [ ] 这些教训是否已应用？
  - [ ] 我的运维方法是否在改进？
  - [ ] 是否有我应该更新的知识？

- [ ] **下一步 (Next Steps)**
  - [ ] 最关键的运维行动项是什么？
  - [ ] 应该与谁讨论这些改进？
  - [ ] 应该何时开始实施改进？
  - [ ] 应该如何衡量成功？

## 检查清单 (Checklist)

在完成这个思维练习后，确认：

- [ ] 进行了全面的基础设施和部署评估
- [ ] 识别了 3-5 个关键的可靠性风险
- [ ] 识别了 3-5 个改进机会（可靠性、速度、安全、成本）
- [ ] 制定了自动化优先级列表
- [ ] 制定了成本优化计划
- [ ] 评估了安全态势和改进需求
- [ ] 规划了改进的实施步骤

---

## 实施与跟踪 (Implementation & Tracking)

- **频率 (Frequency)** - 每月进行一次全面评估，特别是在重大事故或部署问题后进行加强
- **沟通 (Communication)** - 定期与工程和产品团队分享可靠性和部署指标
- **行动 (Action)** - 将改进转化为具体的任务或项目
- **追踪 (Tracking)** - 定期监控关键指标（正常运行时间、部署速度、成本、安全）
