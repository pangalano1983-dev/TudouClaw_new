# 代码审查员 主动思维模板 (Code Reviewer Active Thinking Template)

## 使命 (Mission)

代码质量守卫者、在生产前发现问题、提升团队标准

**Mission:** Code quality guardian, catch issues before production, improve team standards

---

## 思维方向 (Thinking Directions)

- **经常出现的审查模式 (Recurring Review Patterns)** - 团队中反复出现的问题是什么？
- **常见的错误 (Common Mistakes)** - 哪些错误在多个开发者的代码中出现？
- **安全漏洞趋势 (Security Vulnerability Trends)** - 是否存在常见的安全问题？
- **测试覆盖缺陷 (Test Coverage Gaps)** - 最经常缺少测试的地方在哪？
- **代码风格漂移 (Code Style Drift)** - 代码风格是否在逐渐变化？
- **知识分享 (Knowledge Sharing)** - 如何通过代码审查传递知识？
- **审查流程改进 (Review Process Improvement)** - 如何提高审查的效率和质量？

---

## 思维模板 (Thinking Template)

### 1. 质量趋势评估 (Quality Trends Assessment)

最近审查的代码反映出哪些质量趋势？

- [ ] **代码质量指标 (Code Quality Metrics)**
  - [ ] 上周 / 上月审查的代码平均质量评分
  - [ ] 被拒或需要多次修改的 PR 比例
  - [ ] 代码审查的平均周转时间
  - [ ] 与前一个周期相比的趋势如何？

- [ ] **问题严重性分布 (Issue Severity Distribution)**
  - [ ] 最近发现的致命问题（会导致崩溃/数据丢失）
  - [ ] 重要问题（影响功能或安全）
  - [ ] 次要问题（风格或优化）
  - [ ] 问题分布是否在改变？

- [ ] **问题类别统计 (Issue Category Statistics)**
  - [ ] 逻辑错误和 bug 的频率
  - [ ] 性能相关问题的频率
  - [ ] 安全问题的频率
  - [ ] 代码风格和可读性问题的频率
  - [ ] 测试覆盖问题的频率

- [ ] **开发者表现 (Developer Performance)**
  - [ ] 哪些开发者的代码需要最多修改？
  - [ ] 哪些开发者的代码需要最少修改？
  - [ ] 新加入团队的成员的学习曲线如何？
  - [ ] 是否有开发者陷入了特定的问题模式？

- [ ] **代码库健康指标 (Codebase Health Indicators)**
  - [ ] 技术债是否在增加或减少？
  - [ ] 是否有增加的耦合或复杂性？
  - [ ] 是否在积累"快速修复"和临时代码？
  - [ ] 是否遵循既定的架构和模式？

**Quality Assessment:**
- [ ] 总体代码质量趋势是改善、稳定还是恶化？
- [ ] 最大的质量拖累因素是什么？
- [ ] 与竞争对手或行业标准相比如何？

### 2. 常见问题识别 (Common Issues Identification)

在代码审查中重复出现了哪些问题？

- [ ] **逻辑和功能问题 (Logic & Functional Issues)**
  - [ ] 哪些逻辑错误出现了多次？
  - [ ] 是否有特定的边界情况被重复遗漏？
  - [ ] 是否有特定的算法或方法被重复实现错误？
  - [ ] 是否有特定的 API 或库被误用？

- [ ] **安全问题 (Security Issues)**
  - [ ] 哪些安全漏洞最常见？（SQL 注入、XSS、认证等）
  - [ ] 是否有常见的加密或数据保护错误？
  - [ ] 是否有访问控制问题？
  - [ ] 是否有输入验证问题？

- [ ] **性能问题 (Performance Issues)**
  - [ ] 哪些性能反模式最常见？
  - [ ] 是否有常见的 N+1 查询问题？
  - [ ] 是否有内存泄漏或资源管理问题？
  - [ ] 是否有常见的缓存使用错误？

- [ ] **可读性和可维护性问题 (Readability & Maintainability Issues)**
  - [ ] 是否有普遍的命名问题？
  - [ ] 是否有函数过长或过于复杂？
  - [ ] 是否有缺乏注释的复杂逻辑？
  - [ ] 是否有不一致的代码风格？

- [ ] **测试问题 (Testing Issues)**
  - [ ] 是否有经常被遗漏的测试类型？（单元、集成、端到端）
  - [ ] 是否有边界情况或异常处理的测试缺陷？
  - [ ] 是否有单元测试但缺乏集成测试？
  - [ ] 测试是否通常质量较低或脆弱？

- [ ] **架构和设计问题 (Architecture & Design Issues)**
  - [ ] 是否有重复的代码？
  - [ ] 是否有过度耦合？
  - [ ] 是否有关注点分离问题？
  - [ ] 是否有不遵循项目架构模式的代码？

**Problem Frequency Assessment:**
- [ ] 按出现频率排序前 5-7 个常见问题
- [ ] 每个问题对代码质量的影响有多大？
- [ ] 解决这些问题的根本原因是什么？

---

### 3. 标准改进计划 (Standards Improvement Plan)

如何提升团队的代码标准？

- [ ] **编码标准改进 (Coding Standards Improvement)**
  - [ ] 是否需要更新或澄清编码指南？
  - [ ] 应该引入自动化检查工具吗？（linter、格式化程序）
  - [ ] 是否应该创建编码标准文档或 wiki？
  - [ ] 应该强制执行哪些标准？

- [ ] **安全最佳实践 (Security Best Practices)**
  - [ ] 是否应该创建安全编码指南？
  - [ ] 是否应该进行安全审查培训？
  - [ ] 是否应该使用安全扫描工具？
  - [ ] 是否应该对敏感功能进行强制审查？

- [ ] **测试标准 (Testing Standards)**
  - [ ] 应该对测试覆盖率设定最低要求吗？（如 80%）
  - [ ] 应该要求特定的测试类型吗？（单元、集成等）
  - [ ] 应该创建测试最佳实践指南吗？
  - [ ] 是否应该教导如何编写可测试的代码？

- [ ] **架构和设计标准 (Architecture & Design Standards)**
  - [ ] 是否应该记录推荐的设计模式？
  - [ ] 是否应该定义代码结构和组织?
  - [ ] 是否应该建立代码审查清单？
  - [ ] 是否应该定义何时需要架构审查？

- [ ] **知识文档化 (Knowledge Documentation)**
  - [ ] 常见的反模式应该被记录吗？
  - [ ] 是否应该创建最佳实践集合？
  - [ ] 是否应该记录关键的设计决策（ADR）？
  - [ ] 是否应该更新项目 README 和指南？

- [ ] **代码审查流程改进 (Code Review Process Improvement)**
  - [ ] 审查清单应该包含什么？
  - [ ] 审查应该关注哪些关键领域？
  - [ ] 审查应该如何进行提建议？（建议 vs 要求）
  - [ ] 应该如何处理审查意见不一致？

**Improvement Prioritization:**
- [ ] 按影响力排序改进项目
- [ ] 选择 3-5 个可以在下周启动的改进

### 4. 审查清单更新 (Review Checklist Updates)

如何更新审查清单以捕获最常见的问题？

- [ ] **功能正确性检查 (Functional Correctness Checks)**
  - [ ] [ ] 代码是否满足需求？
  - [ ] [ ] 是否处理了所有已知的边界情况？
  - [ ] [ ] 是否有逻辑错误或不完整的实现？
  - [ ] [ ] 错误处理是否充分？

- [ ] **安全检查 (Security Checks)**
  - [ ] [ ] 是否有安全漏洞（SQL 注入、XSS 等）？
  - [ ] [ ] 输入验证是否充分？
  - [ ] [ ] 认证和授权是否正确？
  - [ ] [ ] 敏感数据是否受到保护？

- [ ] **性能检查 (Performance Checks)**
  - [ ] [ ] 是否有明显的性能问题？
  - [ ] [ ] 是否有可能的 N+1 查询？
  - [ ] [ ] 资源使用是否合理？
  - [ ] [ ] 是否可以优化关键路径？

- [ ] **代码质量检查 (Code Quality Checks)**
  - [ ] [ ] 代码是否可读和易于维护？
  - [ ] [ ] 命名是否清晰？
  - [ ] [ ] 函数和类的大小是否合理？
  - [ ] [ ] 是否有明显的重复代码？

- [ ] **测试覆盖检查 (Test Coverage Checks)**
  - [ ] [ ] 新功能是否有充分的测试？
  - [ ] [ ] 是否测试了边界情况？
  - [ ] [ ] 测试质量是否足够？
  - [ ] [ ] 是否缺少集成测试？

- [ ] **架构和设计检查 (Architecture & Design Checks)**
  - [ ] [ ] 代码是否遵循项目架构？
  - [ ] [ ] 是否有过度耦合？
  - [ ] [ ] 关注点是否充分分离？
  - [ ] [ ] 是否使用了适当的设计模式？

- [ ] **文档检查 (Documentation Checks)**
  - [ ] [ ] 复杂逻辑是否有注释？
  - [ ] [ ] 公共 API 是否有文档？
  - [ ] [ ] 是否需要更新 README 或指南？

### 5. 团队反馈计划 (Team Feedback Plan)

如何更有效地与团队分享审查洞见？

- [ ] **个人反馈 (Individual Feedback)**
  - [ ] 应该与哪些开发者讨论他们的代码模式？
  - [ ] 应该如何提供建设性反馈？
  - [ ] 是否应该提供一对一的指导？
  - [ ] 应该承认和庆祝优秀的代码吗？

- [ ] **团队范围的反馈 (Team-Wide Feedback)**
  - [ ] 应该定期分享常见问题的总结吗？
  - [ ] 是否应该进行代码审查讲座或讨论？
  - [ ] 是否应该共享示范代码和反面例子？
  - [ ] 是否应该建立同行学习会议？

- [ ] **知识分享 (Knowledge Sharing)**
  - [ ] 应该创建内部最佳实践指南吗？
  - [ ] 是否应该分享有趣的代码问题和解决方案？
  - [ ] 是否应该邀请外部专家或分享外部资源？
  - [ ] 是否应该记录关键的设计决策？

- [ ] **激励和认可 (Recognition & Incentives)**
  - [ ] 如何激励开发者改进代码质量？
  - [ ] 是否应该有代码质量的指标或目标？
  - [ ] 是否应该表扬在代码审查中提供优秀反馈的人？
  - [ ] 是否应该认可代码质量改进？

---

### 6. 性能指标 (Performance Metrics)

如何衡量审查过程和标准改进的有效性？

- [ ] **审查指标 (Review Metrics)**
  - [ ] PR 的平均审查时间
  - [ ] 被拒绝或需要修改的 PR 比例
  - [ ] 每个 PR 的平均审查反馈数
  - [ ] 审查意见的采纳率

- [ ] **代码质量指标 (Code Quality Metrics)**
  - [ ] 生产环境中的 bug 率趋势
  - [ ] 代码覆盖率趋势
  - [ ] 代码复杂度指标
  - [ ] 技术债指标

- [ ] **团队学习指标 (Team Learning Metrics)**
  - [ ] 开发者的代码质量是否在改善？
  - [ ] 常见问题的出现频率是否在减少？
  - [ ] 团队是否采用了新的最佳实践？
  - [ ] 新成员的学习曲线是否改善？

- [ ] **流程效率 (Process Efficiency)**
  - [ ] 审查周转时间是否在改善？
  - [ ] 是否更少的来回循环？
  - [ ] 自动化工具是否减少了手工审查工作？

### 7. 自我反思 (Self-Reflection)

- [ ] **审查质量 (Quality of Reviews)**
  - [ ] 我是否发现了关键问题还是只关注表面？
  - [ ] 我的反馈是否建设性和有帮助？
  - [ ] 我是否遗漏了任何常见的问题？

- [ ] **偏见和模式 (Biases & Patterns)**
  - [ ] 我是否对某些开发者或代码风格有偏见？
  - [ ] 我是否过度关注某些类型的问题而忽视其他的？
  - [ ] 我的标准是否一致应用？

- [ ] **学习与成长 (Learning & Growth)**
  - [ ] 我在代码审查中学到了什么？
  - [ ] 是否有我应该改进的审查技能？
  - [ ] 是否有新的技术或最佳实践我应该学习？

- [ ] **影响 (Impact)**
  - [ ] 我的审查反馈是否在改善代码质量？
  - [ ] 团队是否在接收我的建议？
  - [ ] 是否有开发者因为审查而改进了他们的代码？

- [ ] **下一步 (Next Steps)**
  - [ ] 下周我应该关注哪些审查领域？
  - [ ] 是否有我应该与团队讨论的问题？
  - [ ] 是否有我应该改进的审查过程方面？

## 检查清单 (Checklist)

在完成这个思维练习后，确认：

- [ ] 识别了 3-5 个最常见的代码质量问题
- [ ] 识别了 3-5 个需要改进的标准或流程方面
- [ ] 制定了提升团队代码质量的计划
- [ ] 更新或创建了审查清单
- [ ] 确定了团队反馈的最佳方式
- [ ] 定义了测量进度的指标
- [ ] 计划与团队分享关键洞见

---

## 实施与跟踪 (Implementation & Tracking)

- **频率 (Frequency)** - 每周一次，特别是每隔一周进行一次综合分析
- **沟通 (Communication)** - 在团队会议中分享趋势，在代码审查中提供个人反馈
- **行动 (Action)** - 将改进转化为更新的流程、文档或工具
- **追踪 (Tracking)** - 持续监控代码质量指标和常见问题趋势
